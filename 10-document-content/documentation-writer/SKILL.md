# Documentation Writer

## Goal
Create concise, accurate, implementation-aligned technical documentation.

## When to use
- Writing README files or quickstarts
- Creating architecture documentation
- Building setup or contributor guides
- Improving existing documentation
- Documenting APIs or interfaces
- Summarizing a release, changelog entry, or version update
- Drafting user-facing or developer-facing release notes
- Converting a feature idea into structured requirements
- Creating scope documents, feature specs, or PRDs

## When not to use
- Marketing copy or sales materials
- Legal documents or compliance docs
- User-facing product copy
- Direct code generation or implementation

## Process

### Technical Documentation
1. Identify audience and purpose.
2. Extract only verified behavior.
3. Organize from most useful to least useful.
4. Prefer examples over abstract explanation.
5. Avoid unsupported claims.
6. Include troubleshooting if relevant.

### Release Notes
1. Identify version and audience.
2. Group changes by theme.
3. Highlight user-visible and developer-visible impact separately.
4. Avoid vague hype language.
5. Mention fixes, breaking changes, and upgrade notes.
6. Keep claims aligned with verified changes only.

### PRD Writing
1. Restate the problem in one sentence.
2. Identify target users and pain points.
3. Define outcomes and constraints.
4. Split requirements into must-have, should-have, and out-of-scope.
5. Add user stories and success metrics.
6. End with acceptance criteria and risks.

## Output formats

### Technical Documentation
- Audience
- Purpose
- Prerequisites
- Steps
- Examples
- Troubleshooting
- Limitations

### Release Notes
- Version
- Highlights
- Fixes
- Breaking changes
- Upgrade notes
- Known limitations

### PRD
- Problem
- Users
- Goals
- Requirements
- Non-goals
- User stories
- Metrics
- Risks
- Acceptance criteria
